# Участие в проекте

Спасибо за интерес к улучшению базы знаний Cyrillic Languages. Этот документ объясняет, что можно править контрибьюторам, описывает формат файлов данных, синтаксис маркеров глифов и порядок отправки Pull Request.

Англоязычная версия — [CONTRIBUTING.md](CONTRIBUTING.md).

## Для кого эта инструкция

Руководство предназначено всем, кто хочет исправить или дополнить информацию о **кириллическом языке**: поправить алфавит, обновить описание, добавить диалектную форму, уточнить альтернативные названия и так далее.

В репозитории есть также директория с латиницей (`cyrillic-languages/library/latin/`), но это внутренняя работа в процессе — **для внешних PR она закрыта**. Пожалуйста, не отправляйте PR против неё.

## Что можно редактировать

Контрибьюторы должны менять файлы **только** внутри:

```
cyrillic-languages/library/cyrillic/base/
```

Любая другая директория — включая `site/`, `static/`, `scripts/` и общие конфигурационные файлы рядом с ними — поддерживается кураторами проекта, и трогать её в PR контрибьютора не следует. Полный список путей — в разделе **Repository map** корневого [README.md](README.md).

Простое правило: если вы меняете один JSON-файл одного языка — всё хорошо.

## Процесс

1. **Форкните** репозиторий.
2. **Отредактируйте** один или несколько `.json` файлов в `cyrillic-languages/library/cyrillic/base/`.
3. **Откройте Pull Request** против ветки `master`.
4. Куратор проверит изменение, при необходимости попросит правок и — после принятия — отдельным коммитом пересоберёт артефакты сайта в `cyrillic-languages/site/`. Вам не нужно пересобирать и коммитить ничего под `site/` самостоятельно.

Заголовок и описание PR можно писать **на русском или английском** — как удобнее. Пожалуйста, кратко поясните, *почему* вносится изменение, и, где применимо, укажите источник (словарь, орфографический справочник, академическая публикация, консенсус в языковом сообществе…), на который вы опираетесь.

## JSON-файл одного языка

Один файл на один язык, с именем по английскому названию, например `Avar.json`. Имя файла должно точно совпадать со значением поля `name_eng` внутри.

### Обязательные поля

| Поле | Тип | Описание |
|---|---|---|
| `name_eng` | строка | Английское название языка. Должно совпадать с именем файла (без `.json`). |
| `name_rus` | строка | Русское название языка. |
| `local` | строка | OpenType-тег локали, по которому включаются локализованные формы глифов. Разрешённые значения: `ru` (по умолчанию), `ba` (башкирский), `bg` (болгарский), `cv` (чувашский), `sr` (сербский). Если вашему языку нужна новая локаль — откройте issue, это требует изменений в коде. |
| `language_group_rus` | массив строк | Классификация языковой семьи по-русски, от более широкой к более узкой. Пример: `["Индоевропейские языки", "Славянские языки"]`. |
| `alt_names_eng` | массив строк | Альтернативные / исторические английские названия языка. Всегда список, даже если в нём одно название. |
| `description_rus` | строка | Описание языка и его алфавита простым текстом, по-русски. |
| `glyphs_list` | массив | Алфавит и связанные наборы знаков. См. раздел [Блоки глифов](#блоки-глифов) ниже. |

### Рекомендуемые поля

| Поле | Тип | Описание |
|---|---|---|
| `description_eng` | строка | Описание простым текстом по-английски. Примерно у 40% существующих файлов это поле сейчас пустое — в новых правках его стоит заполнять. |
| `language_group_eng` | массив строк | Английский аналог `language_group_rus`. |

### Форматирование текста

`description_*`, `name_*`, `language_group_*` и `alt_names_eng` — это **простой текст**, без Markdown и HTML. Внутри `description_*` можно использовать `\n` для разделения абзацев. Завершающий `\n` не требуется. Типичное описание — один-два абзаца (до ~500 символов).

### Внутренние / игнорируемые поля

В некоторых файлах могут встречаться поля, которые контрибьюторам добавлять и менять не нужно:

- `code_pt` (находится в `cyrillic_library.json`, а не внутри файла языка) — внутренний идентификатор Paratype. Реальное значение назначают кураторы.

## Блоки глифов

Поле `glyphs_list` — это массив блоков. Каждый блок — отдельное представление инвентаря глифов языка: собственно алфавит, исторические формы, диалектные формы и т.д.

Блок выглядит так:

```json
{
    "type": "alphabet",
    "uppercase": "А Б В Г ...",
    "lowercase": "а б в г ..."
}
```

### Типы блоков

| `type` | Значение |
|---|---|
| `alphabet` | Действующий официальный алфавит языка. Во всех файлах ровно один блок `alphabet`. |
| `extended` | Расширенная орфографическая нотация — знаки, используемые вместе с алфавитом (например, кодпоинты из зоны Private Use Area, отрисовываемые шрифтами Paratype, дополнительные буквы для определённых контекстов). |
| `historic` | Исторические формы, не входящие в современный алфавит, но полезные в исторических текстах и справочных материалах. |
| `dialect` | Формы глифов, используемые в диалектах языка и не входящие в стандартный алфавит. |
| `consideration` | Знаки, по которым **в самом языковом сообществе пока нет консенсуса** — например, символы, использование которых активно обсуждается носителями или специалистами. Используйте этот тип осторожно: оценочное суждение здесь принадлежит людям, знающим язык, а не пайплайну. |

Блок можно опустить, если в нём нечего размещать — не у каждого языка есть `historic` или `dialect`.

### Строки `uppercase` и `lowercase`

Обе строки — это списки **токенов глифов**, разделённые пробелами. Обычно токен — это один глиф (один Unicode-символ), но также бывает:

- **Многосимвольный диграф** (например, `Дж`, `Дз` в белорусском).
- **Escape-последовательность Unicode** вида `!F86F` для символа, который невозможно набрать с клавиатуры — как правило, кодпоинт из зоны Private Use Area, отрисовываемый шрифтами Paratype.
- Любой из вышеперечисленных вариантов с **одним или несколькими маркерами** перед или после — для дополнительной семантики; см. следующий раздел.

Обе строки должны идти параллельно: каждому глифу в верхнем регистре соответствует свой собрат в нижнем регистре на той же позиции, и наоборот.

## Маркеры глифов — живые маркеры

Маркеры — это одиночные символы или суффиксы, приписанные к токену **без пробела между маркером и токеном**. Их интерпретирует пайплайн компиляции. На один токен можно навесить несколько маркеров; порядок их записи неважен.

### `+` — Альтернатива со своим Unicode

Группирует токен под **предыдущим** основным глифом как альтернативную форму. У альтернативы есть собственный корректный Unicode-кодпоинт.

Пример — фрагмент блока `alphabet` из [Bashkir.json](cyrillic-languages/library/cyrillic/base/Bashkir.json):

```
А Б В Г &!F53C +Ғ Д &!F50A +Ҙ Е Ё ... С &!F53E +Ҫ Т У Ү Ф Х Һ ...
```

Каждая `+<буква>` привязывается к токену, стоящему непосредственно перед ней. В паре `&!F53C +Ғ` основной глиф — кодпоинт из зоны Private Use Area (см. раздел `!XXXX` ниже), отображаемый через OpenType-фичу `locl` (маркер `&`), а `Ғ` — альтернативная форма с собственным стандартным Unicode.

### `&` — Локализованная форма

Помечает токен как глиф, который должен отрисовываться через OpenType-фичу `locl` для значения `local` данного языка. Обычно используется вместе с `+` или `!XXXX`. Тот же фрагмент из [Bashkir.json](cyrillic-languages/library/cyrillic/base/Bashkir.json) иллюстрирует маркер: `&!F53C`, `&!F50A`, `&!F53E` — все это кодпоинты в зоне Private Use Area (см. заметку про PUA в разделе `!XXXX` ниже), которые шрифты Paratype подставляют, когда активна `locl` для башкирского (`"local": "ba"`).

### `:` — Диграф

Группирует токен под предыдущим глифом как диграф — последовательность из двух символов, которая рассматривается как одна буква алфавита.

Пример — фрагмент блока `alphabet` из [Belarusian.json](cyrillic-languages/library/cyrillic/base/Belarusian.json):

```
А Б В Г Ґ Д :Дж :Дз Е Ё Ж З І Й К Л М Н О П Р С Т У Ў Ф Х Ц Ч Ш Ы Ь Э Ю Я ʼ
```

`Дж` и `Дз` — белорусские диграфы, привязанные к `Д`.

### `!XXXX` — Escape-последовательность Unicode

Литерал Unicode-кодпоинта в виде hex-кода. Используется, когда символ нельзя удобно набрать с клавиатуры — на практике это почти всегда кодпоинт из зоны Private Use Area (PUA, `E000`–`F8FF`).

Пример — фрагмент блока `extended` из [Avar.json](cyrillic-languages/library/cyrillic/base/Avar.json):

```
!F86F !F872 !F875 !F878 !F87B !F87E
```

Можно сцеплять: `!F86F!F870` даёт двухсимвольную последовательность.

#### Про зону Private Use Area

PUA — это диапазон кодпоинтов, который консорциум Unicode намеренно оставляет неназначенным: чистый холст, которым любой шрифт или приложение могут распорядиться по собственному усмотрению. **Эти кодпоинты никому не принадлежат.** Paratype исторически использует часть этого диапазона (ориентировочно от `F000` и выше) в своих собственных шрифтах, чтобы закодировать глифы, у которых нет официального Unicode-кодпоинта — например, исторические или диалектные кириллические буквы. Это сопоставление имеет смысл **только** тогда, когда текст отрисовывается в шрифте Paratype, в котором эти глифы действительно есть — PT Sans Expert / PT Serif Expert, которые поставляются с этим проектом (см. [README](README.md)).

**В шрифте другой студии тот же hex-код `!FXXX` обычно отобразит совершенно другой глиф — или ничего (пустой квадратик).** Читайте `!FXXX` в этом репозитории как *«глиф по этому кодпоинту в шрифтах Paratype Expert»*, а не как универсальную ссылку на символ.

**Не добавляйте новые `!FXXX` в PR контрибьютора.** Хотя сама зона PUA ничья, осмысленная запись здесь требует, чтобы этот кодпоинт был фактически назначен глифу внутри шрифта Paratype — это согласованное изменение на стороне шрифтов. Если вам кажется, что нужен глиф без официального Unicode — откройте issue.

### `.ita` / `.str` — Стиле-специфичные глифы

Суффиксы. `.ita` помечает глиф, который должен появляться только в курсивном начертании; `.str` (от английского "straight", прямое) — глиф, появляющийся только в прямом начертании. Применяется, когда у одной и той же буквы два разных дизайна, которые выбираются начертанием, а не локалью. Сейчас используется только в **болгарском** и **сербском**.

Пример — фрагмент нижнего регистра алфавита из [Bulgarian.json](cyrillic-languages/library/cyrillic/base/Bulgarian.json):

```
... &в +в &г.str +г.ita &д +д е &ж +ж &з +з &и.str +и.ita ...
```

Здесь `&г.str` — локализованная болгарская форма `г` для прямого начертания, а `+г.ita` — альтернативная форма для курсива.

### Комбинирование маркеров

Маркеры можно навешивать на один токен стопкой. Порядок записи не влияет на парсинг. Например, `&!F53C` и `!F53C&` будут прочитаны одинаково — как «PUA-кодпоинт, являющийся локализованной формой».

## Устаревшие маркеры — не использовать

В более ранних версиях пайплайна была другая схема. Эти маркеры всё ещё перечислены в коде парсера, но **в данных нигде не используются**, и добавлять их не следует:

- `*` (замещение), `$` (расширение / лексикографика), `#` (исторические формы), `@` (диалекты), `<` (алфавит) — их роль теперь выполняет поле `type` в блоках `glyphs_list`.
- `.alt` — в парсере упомянут, но неактивен; **его использование уронит компилятор.**
- `(`, `)`, `[`, `]` — перечислены в таблице маркеров парсера, но в обработчике их нет; **использование уронит компилятор.**

Если вы видите один из этих маркеров в существующем файле, это почти наверняка старый пережиток — отметьте его в PR, но не копируйте паттерн.

## Правка существующего языка

Обычный случай — добавить, убрать или исправить глиф в существующем алфавите либо обновить описание.

1. Откройте `cyrillic-languages/library/cyrillic/base/<Language>.json`.
2. Найдите нужное поле — как правило, блок глифов внутри `glyphs_list` или одно из описаний.
3. Сделайте минимальное необходимое изменение. Держите строки `uppercase` и `lowercase` в соответствии: на каждой позиции верхнего регистра должен быть свой напарник в нижнем, и наоборот.
4. Сохраните, закоммитьте, откройте PR.

**Если вы добавляете глиф, Unicode-кодпоинта которого ещё нет в таблице сортировки** (`cyrillic-languages/library/cyrillic/sortorder_cyrillic.txt`), явно укажите это в описании PR. Таблица сортировки — зона куратора, но ваше замечание подскажет ему добавить недостающую запись.

## Добавление нового языка

Это более крупное изменение — оно затрагивает несколько файлов и может требовать новых записей в общей таблице сортировки. Прежде чем что-то делать, пожалуйста, **сначала откройте issue** с описанием языка, который хотите добавить — это помогает кураторам спланировать сортировку и назначить `code_pt`.

После обсуждения PR для нового языка обычно содержит:

- Новый файл `cyrillic-languages/library/cyrillic/base/<Language>.json`, заполненный согласно этому документу.
- (Через куратора) соответствующую запись в `cyrillic-languages/library/cyrillic/cyrillic_library.json` с `enable: true` и значением `code_pt`, которое назначит куратор; если вы всё же хотите включить такую запись в PR, используйте заглушку `"code_pt": "1"`.

Контрибьюторам не нужно (и не следует) самостоятельно править `sortorder_cyrillic.txt`.

## Чек-лист перед отправкой

Перед открытием PR, пожалуйста, проверьте:

- [ ] Вы редактировали только файлы внутри `cyrillic-languages/library/cyrillic/base/`.
- [ ] Вы **не** меняли ничего в `site/`, `static/`, `scripts/`, `fonts/` и других общих конфигурационных файлах.
- [ ] Имя файла в точности совпадает с полем `name_eng`.
- [ ] Все обязательные поля присутствуют и не пустые.
- [ ] `local` — одно из разрешённых значений (`ru`, `ba`, `bg`, `cv`, `sr`).
- [ ] В токенах глифов используются только живые маркеры, описанные выше. Никаких `*`, `$`, `#`, `@`, `<`, `.alt`, `(`, `)`, `[`, `]`.
- [ ] Вы не вводили новых кодпоинтов из зоны Private Use Area (`!FXXX`). Если PUA был нужен — вместо правки вы открыли issue; добавление PUA-записи требует согласованного изменения в шрифтах Paratype.
- [ ] Если добавляется символ, Unicode которого пока нет в `sortorder_cyrillic.txt`, это упомянуто в описании PR.
- [ ] Описание PR объясняет, что изменено, и, где применимо, ссылается на источник.

## Если что-то неясно

Если какой-то момент этого документа непонятен, или вы не уверены, подходит ли ваше изменение для PR контрибьютора или требует действия куратора, — пожалуйста, откройте issue. Лучше спросить, чем гадать.

Контакт для вопросов вне репозитория: **info@paratype.net**.
